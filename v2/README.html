Sermon and Reference Assistant - v2.0 rewrite plan

HOPTranscribe v2

Realtime sermon transcription with AI-powered Bible scripture suggestions, rebuilt around a split-stage pipeline: dedicated streaming STT for speech and a separate text LLM for scripture matching.

1. Why a v2 rewrite

The v1 implementation combines transcription and scripture matching inside a single OpenAI Realtime voice-agent call. v2 separates the responsibilities, removes unused voice output, and fixes production-readiness gaps.

  1. Cost. v1 pays premium audio token rates for a workload that never needs voice output.
  2. Latency. The realtime voice model has to decide between speaking, calling a tool, and staying silent, adding avoidable delay.
  3. Accuracy. One model is doing passive listening, transcription, and ranked scripture matching mid-stream, which increases false positives.
  4. Operational risk. In-memory production sessions, insecure WebSocket auth, unauthenticated SignalR, and a latent JSON parse bug need to be removed.
v2 splits the pipeline, uses transcription-only STT, moves matching server-side with structured outputs, persists sessions, and locks down session ownership.

5. AI pipeline detail

5.1 Transcription

Browser creates a WebRTC peer connection, adds the mic track, opens an oai-events data channel, posts the SDP offer to OpenAI with an ephemeral bearer token, and consumes delta/completed events.

  • Endpoint: /v1/realtime/transcription_sessions
  • Model: gpt-realtime-whisper
  • Turn detection: server VAD
  • No audio output or TTS modality

5.2 Scripture Matching

Matching runs server-side only. The browser never receives the system prompt or verse validation data.

  • Endpoint: POST /api/match
  • Model: gpt-5-mini, fallback gpt-4o-mini
  • Output: strict JSON schema
  • Validation: canonical book, chapter, verse range, version

11. Phased delivery plan

Each phase is independently demoable and gated by passing CI.

0

Scaffold

  • Create the v2 skeleton, archive v1, initialize backend, frontend, Docker Compose, and CI build gates.
1

Auth and sessions

  • Add JWT claim/refresh, SQLite sessions, CRUD endpoints, ownership rules, and persistence tests.
2

STT pipeline

  • Create transcription sessions, connect WebRTC, and render live transcript without scripture matching.
3

Matching service

  • Call gpt-5-mini, enforce JSON schema, validate scripture references, add rate limiting, and test edge cases.
4

End-to-end wiring

  • Call matching on finalized utterances, persist transcript segments, and render scripture cards.
5

Collaboration

  • Authorize SignalR, join session groups, create read-only viewer flow, and test two-client sync.
6

Polish and hardening

  • Add middleware, error boundaries, toasts, reconnect backoff, settings, telemetry, logging, and load tests.
7

Azure deploy

  • Build Container Apps Terraform, GitHub Actions deployment, ACR publish, and smoke tests.

13. Deployment

Production target is Azure Container Apps: separate api and web apps behind a shared environment, Log Analytics, Application Insights, and persistent storage for SQLite.

Storage

SQLite persists on a Container Apps storage mount backed by Azure Files. Single-replica is acceptable for v2.

Secrets

OpenAI__ApiKey and Jwt__SigningKey live in Container Apps secrets or Key Vault references.

Ingress

The web app is external. The API can be internal or external behind APIM, with custom domains via managed certs.

Terraform
cd v2/infra/azure
terraform init
terraform apply \
  -var "openai_api_key=$OPENAI_API_KEY" \
  -var "jwt_signing_key=$JWT_SIGNING_KEY"
